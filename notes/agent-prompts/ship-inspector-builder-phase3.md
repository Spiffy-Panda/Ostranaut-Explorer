## Goal

Build a new self-contained page at
`src/Ostranauts.Site/ship-inspector.html` plus its companion
`src/Ostranauts.Site/ship-inspector.js`. The page lets a modder pick a
ship via dropdown (canned set) or upload one of their own, and shows:

- A coarse 2D wall/floor/empty grid of the ship's tiles.
- A foldable component checklist grouped by bucket
  (walls/floors/doors/conduits/containers/equipment/decorative/other),
  with a checkbox per component-instance and per-bucket `<checked>/<total>`
  counters.
- Per-room requirement cards (using `window.renderRoomCard` from Phase 2),
  pinnable to a top section.
- A sticky overall-progress bar at the top of the components section.
- localStorage-persisted state per ship (checkbox state, pinned rooms,
  upload cache).

Plain vanilla JS, no build framework — must run from a `file://` URL.

## Your deliverable — Phase 3

Two new files plus loader plumbing.

### `src/Ostranauts.Site/ship-inspector.html`

A self-contained HTML page with the layout described in PLAN-BUILDER.md's
Phase 3 section (block diagram). Loads:

- `theme-toggle.js` (header dark/light toggle parity with other pages)
- `rooms.js` (Phase 2 shared module — gives you `window.ROOMS` and
  `window.renderRoomCard`)
- `ship-inspector.js` (your new module — see below)
- `style.css` (existing site stylesheet — extend with inspector-specific
  rules at the bottom or in a small page-local `<style>` block; reuse
  tokens from the `:root` palette)

The friendly-name map and the manifest can be loaded via `fetch()`
since the page targets `file://` plus the preview server — but `fetch()`
with `file://` is blocked in some browsers, so prefer the
`<script src="data/id-friendly-names.js">` pattern used elsewhere in
the site. Phase 1 didn't emit a `.js` wrapper — a small additional
build step in Phase 3 might be: at page load, load the JSONs as
`<script>` tags that re-emit them as `window.SHIP_FRIENDLY_NAMES =
{...}` etc. The cleanest approach is probably to **add a lightweight
JS-wrapper companion** to the JSON outputs — either by modifying
`utils/python/build_ship_inspector_data.py` to emit both `.json` and
`.js`, or by adding a tiny new emitter step. **Both are acceptable;
pick one and document it in your DEV-LOG entry.** Don't move the JSONs
themselves out of the way — modders are encouraged to read the
friendly-name map directly, and the canned-ship JSONs are documented
as build artifacts at the JSON path.

The canned-ship dropdown drives `JSON.parse` of the per-ship file via
the same `<script>`-vs-`fetch` path; uploaded ships go through a file
input + `FileReader.readAsText` then `JSON.parse`. Validate that the
upload looks ship-shaped (single-element array, dict with `aItems`)
before consuming; show a friendly error otherwise.

### `src/Ostranauts.Site/ship-inspector.js`

Your new module. Suggested structure (not prescriptive):

- A small state object `{ activeShipKey, shipData, friendlyNames, pinnedRooms, checkboxState, ... }`.
- Render helpers per layout section: `renderHeader`, `renderShipPicker`,
  `renderProgressBar`, `renderVisualLayout`, `renderPinnedRooms`,
  `renderComponentList`, `renderRoomList`. Each takes the state and
  writes into a known root element.
- A `loadShip(key)` that pulls the ship data, sets `activeShipKey`,
  rehydrates state from localStorage, and re-renders.
- An `onPinRoom(spec)` callback wired into `renderRoomCard` via the
  `onPin` option.

localStorage keys (see PLAN-BUILDER.md table):

| Key | Value |
|---|---|
| `ship-inspector:active-ship` | `<reg>` or `upload:<sha256>` |
| `ship-inspector:checkboxes:<key>` | `{ "<co-instance-id>": true, ... }` |
| `ship-inspector:pinned-rooms:<key>` | `[ "<room-instance-id>", ... ]` |
| `ship-inspector:upload-cache:<sha>` | gzip-compressed uploaded JSON (lz-string or pako) |
| `ship-inspector:upload-meta:<sha>` | `{ name, dateUploaded, summary }` |

For upload compression, **prefer lz-string** (small, single-file, no
WASM, runs from `file://`) over pako. Vendored copy goes in
`src/Ostranauts.Site/vendor/lz-string.min.js` if not already present;
update the page's script tags. If you decide to skip compression for
v1 and just refuse uploads larger than the localStorage budget, that's
acceptable — flag it in your DEV-LOG and Phase 4-or-later can revisit.

Component-list ergonomics:

- Per-foldout count `<checked>/<total>` visible even when collapsed
  (put it in the `<summary>`).
- Sticky overall progress bar at the top of the components section.
- **Search box + filter pills** for the component list — pills toggle
  each bucket on/off; search matches free-text against
  `friendlyName(strName) ?? strName` and against the strName itself.
- Per line item: friendly name + quantity + that-many checkboxes (one
  per instance, each tied to the instance's `strID` so checking
  persists). Falls back to raw `strName` for unknown components, with a
  tooltip *"this ID isn't in the shipped name table — likely a mod
  component."*
- A **"reset checklist"** button per ship that wipes only that ship's
  checkbox state from localStorage + re-renders.

URL-shareable state:

- `ship-inspector.html#/ship/<reg>` for canned ships — selecting a
  ship in the dropdown should update `location.hash`; reading
  `location.hash` on load should select that ship.
- `ship-inspector.html` (no hash) for uploaded ships — file content is
  the state and a URL can't carry it.

Print stylesheet (`@media print`):

- Hide ship dropdown, upload control, visual layout, search, filter
  pills.
- Show component checklist + pinned rooms only.
- Preserve checkbox visual state so a printed checklist matches the
  live one.

### Visual layout (the 2D grid)

A coarse 3-color render computed client-side from `aRooms[]` (which
carries tile coords) + `aItems[]` (each item's `fX`/`fY` plus its
bucket). For v1, palette is **wall / floor / empty**:

- A tile is a *wall* if any item with bucket `walls` lands on it.
- Else a *floor* if any item with bucket `floors` lands on it OR if a
  room's tile list claims it.
- Else *empty*.

Render at a coarse pixel-per-tile (≈8–12 px) on a `<canvas>` or
`<div>`-grid. A simple absolute-positioned grid of small `<div>`s is
fine if `<canvas>` adds friction. Don't sprite-render or color-code
by individual bucket — that's deferred to v2.

### Sanity-check before declaring done

- The page loads from `file://` (no `import`/`export`, no top-level
  await, no MIME-blocked module syntax).
- A canned ship from the dropdown renders the visual layout, the
  component buckets with correct counts (cross-check against
  `canned-ships-manifest.json` `bucketCounts`), and the room list.
- Checking a few boxes, reloading the page, and re-selecting the same
  ship shows the checks still ticked.
- Pinning a room from the room list moves it to the *Pinned rooms*
  section; pinning persists across reload; unpinning returns it.
- The deep-link `#/ship/<reg>` selects that ship on load.
- A small modded ship (or a hand-edited canned ship with a fake
  `strName` not in the friendly-name map) renders the raw ID with the
  *"likely from a mod"* tooltip, doesn't crash.
- The print stylesheet hides the chrome and shows just the checklist
  when previewed via the browser's print preview.

Use the preview server (`http://localhost:8765/ship-inspector.html`)
plus `mcp__Claude_Preview__*` tools to verify. The preview serves
`build/`, so either run `make site` between captures or copy the
modified files into `build/` directly. Cache-bust with `?bust=…` on
the URL since the script tags are aggressively cached.
